# ADR-011: Migrations on Deploy

## Status

Accepted — June 2026

## Context

Per ADR-006 and ADR-007, schema changes are generated by Drizzle Kit and applied by `pnpm db:migrate`. We need to decide **when** that command runs in the deploy pipeline: during the Vercel build, in a post-deploy hook, or manually by a human.

The environments to cover:
1. **Local development** — developer runs `pnpm db:migrate` against a personal Neon branch.
2. **Preview deployments** (one per Git branch) — Vercel + Neon branching create an ephemeral DB per PR; pending migrations must apply before traffic hits.
3. **Production** — main branch deploys must apply pending migrations exactly once and in order.

The decision affects deploy reliability, rollback behavior, and where failures surface.

## Decision Drivers

- **Deterministic application** — migrations must run exactly once and in order; no race conditions across concurrent builds.
- **Fail-fast on schema errors** — a broken migration should fail the deploy before the new code starts serving traffic.
- **Visibility** — failures must appear in the existing Vercel deploy logs, not in a separate dashboard.
- **No manual steps** — humans should not be required to run a CLI command for every deploy.
- **Preview parity** — the same mechanism that applies migrations in production must apply them in previews.

## Options Considered

### Option A — Run during Vercel build (`buildCommand` chain)

`package.json`'s `build` script becomes `pnpm db:migrate && next build`. Vercel runs the chain; if `db:migrate` fails, the build fails and the deploy is aborted before any new function is published.

- Migrations apply against the **target environment's** `DATABASE_URL` (preview or production).
- Failure surfaces as a build failure in the Vercel UI.
- Atomic: if the build fails, the previous version keeps serving.

### Option B — Run in a post-deploy hook

Vercel does not provide native post-deploy hooks for application code. Implementing this requires an external GitHub Action or a Vercel Deploy Hook fired by a webhook after the build completes.

- Adds an external moving part.
- New version is already serving when migrations apply — risks running new code against the old schema for a window.
- Logs split between Vercel and GitHub Actions (or wherever the hook runs).

### Option C — Manual via CLI

Developer runs `pnpm db:migrate` against production before pushing.

- Simple to reason about.
- Easy to forget.
- No preview-deployment story.
- Two humans pushing concurrent migrations can race.

### Option D — On-demand via API route

A protected route (`/api/admin/migrate`) runs `migrate()` on demand.

- No build slowdown.
- Requires authentication, rate limiting, and a trigger mechanism.
- Same race risk as manual; new code can serve before the trigger fires.

## Comparative Analysis

| Criterion | Build step | Post-deploy hook | Manual CLI | API route |
|---|---|---|---|---|
| **Deterministic ordering** | **Yes** | Eventual | **No** (human error) | Eventual |
| **Atomic with deploy** | **Yes** | No (race window) | No | No |
| **Failure visibility** | **Vercel build UI** | External tool | Local terminal | App logs |
| **Preview coverage** | **Automatic** | Requires per-env setup | None | Manual trigger |
| **External services** | None | GitHub Actions / webhook | None | None |
| **Race on concurrent deploys** | Vercel serializes per branch | Possible | Possible | Possible |

## Decision

**Run during Vercel build** is selected.

Rationale:

1. **Atomicity with the deploy.** If migrations fail, the new code never serves. The previous version keeps running on the existing schema. This is the only option that prevents the "new code, old schema" window.
2. **Failure surfaces where the developer is already looking.** Vercel's deploy UI is the canonical place to inspect a failed deploy; routing migration failures there avoids a second dashboard to monitor.
3. **Preview deployments get migrations for free.** Vercel's per-branch builds run the same `build` script with the preview's `DATABASE_URL` (pointing at a Neon branch). No extra configuration; previews stay in lockstep with their schema.
4. **No external services or webhooks.** The post-deploy hook option needs GitHub Actions or equivalent — more surface to maintain and another set of credentials.
5. **Concurrent deploys are serialized per branch by Vercel.** Two pushes to `main` queue; the second waits for the first build to finish. The race condition that would affect manual or API-triggered runs doesn't apply.

The manual and API-triggered options remain valid escape hatches for one-off operational migrations that don't belong in a deploy (e.g., backfilling a large table over many hours).

## Consequences

### Positive

- Migrations apply exactly once per deploy, atomic with the code rollout.
- Preview environments stay schema-consistent without extra config.
- Failures block the deploy and appear in the standard build log.
- No external services or workflows to maintain.

### Negative / Trade-offs

- **Build time grows by the migration runtime.** Most migrations are sub-second; large data backfills must not live in the migration step. Mitigation: documented in the `/migration` skill — schema changes only; backfills go through a separate one-off script or cron.
- **`DATABASE_URL` must be writable from Vercel's build environment.** Mitigation: standard; Neon connection strings work transparently from any IP.
- **Rolling back a migration requires a forward migration.** The build step does not run "down" migrations. Mitigation: consistent with ADR-007 (Drizzle Kit doesn't generate down migrations); manual reverse SQL with a later timestamp.
- **`prebuild` could be used instead of chaining in `build`.** Either works; we chain in `build` to keep the dependency explicit and visible to the next reader.

### Revisit Triggers

- A project introduces multi-hour data migrations → split into "schema migration in deploy" + "backfill via cron or one-off script," do not move the entire pipeline.
- Build time becomes dominated by `db:migrate` (rare; would indicate an unrelated problem) → move to a parallel step or a separate workflow.
- A regulated deployment process requires explicit human approval for schema changes → introduce a gated workflow that runs migrations on approval, not on build.
