# ADR-006: ORM / Database Client

## Status

Accepted — June 2026

## Context

Per ADR-002, the database is **Neon Postgres** (serverless, scale-to-zero, branching for previews). We need to choose how the application layer talks to it. The repository layer (`src/repositories/`) is defined as a strict **SQL resolver** in `src/repositories/CLAUDE.md`: it receives parameters, executes a query, returns a typed result. No business logic. That convention already biases us toward thin abstractions over SQL rather than heavy active-record ORMs.

The choice must also account for the runtime profile imposed by previous ADRs:

1. **Serverless functions on Vercel** (ADR-001) — cold-start cost matters, bundle size matters, no long-lived connection pool by default.
2. **Neon scale-to-zero** (ADR-002) — short-lived HTTP queries are preferable to long-lived TCP pools; the Neon serverless driver exists precisely for this.
3. **No RLS** (ADR-002) — authorization lives in the service layer, so the database client does not need to model row-level policies.
4. **Migrations are part of the template** — the `/migration` skill exists; whichever tool we pick must integrate with a versioned migration workflow in `scripts/migrations/`.
5. **TypeScript-first stack** — the rest of the codebase is strict TS with `type` (never `interface`); the ORM's generated types must compose with that.

## Decision Drivers

- **Cold-start and bundle weight** on Vercel functions (matters per request on dormant projects).
- **Edge runtime compatibility** — Better Auth (ADR-004) and middleware run on the edge; the DB client must too.
- **Neon serverless driver support** — first-class HTTP driver integration, not a workaround.
- **Migration story** — generated from schema or hand-written SQL, but versioned and reproducible.
- **Type ergonomics** — inferred types should flow into entities and repository return values with no manual mapping.
- **SQL transparency** — repository code should read like the SQL it produces (per the layer convention).
- **Maturity and longevity** — the template will outlive this decision; the tool must still be alive in 3 years.

## Options Considered

### Option A — Drizzle ORM

TypeScript-first ORM released in 2022. The schema is a TS file; migrations are generated from schema diffs by `drizzle-kit`.

- **Bundle size:** ~7 KB minified, tree-shakeable per query.
- **Driver:** native `@neondatabase/serverless` (HTTP) and `pg` (TCP) adapters.
- **Edge runtime:** supported out of the box.
- **Migrations:** `drizzle-kit generate` writes SQL files to `scripts/migrations/`; `drizzle-kit migrate` applies them. Hand-written SQL is also supported.
- **API style:** query-builder that mirrors SQL (`db.select().from(users).where(eq(users.id, id))`) plus a `sql\`...\`` template for raw queries.
- **Types:** inferred from the schema file with no codegen step (no generated client to keep in sync).
- **Weekly downloads:** ~1.5M and trending up through 2026.
- **License:** Apache 2.0.

### Option B — Prisma

The most widely adopted TypeScript ORM. Schema lives in a `.prisma` DSL; a client is generated by `prisma generate`.

- **Bundle size:** historically heavy (Rust query engine binary). As of 2025, **Driver Adapters** are stable and allow Prisma to use `@neondatabase/serverless` directly, removing the Rust engine from edge bundles. The TypeScript client preview shipped in early 2026 reduces overhead further.
- **Driver:** Prisma + Neon Driver Adapter is the documented serverless path.
- **Edge runtime:** supported via Driver Adapters; requires explicit configuration.
- **Migrations:** `prisma migrate dev` / `prisma migrate deploy` — schema-first, generates SQL automatically.
- **API style:** fluent, active-record-ish (`prisma.user.findUnique({ where: { id } })`). Hides SQL by default; `$queryRaw` for raw escape hatch.
- **Types:** generated client must be re-run on every schema change (`prisma generate`); types are excellent but coupled to the build step.
- **Weekly downloads:** ~3.5M (largest installed base).
- **License:** Apache 2.0.

### Option C — Kysely

A pure TypeScript query builder (no ORM abstractions). Released in 2022.

- **Bundle size:** ~50 KB, tree-shakeable.
- **Driver:** dialect plugins; Neon serverless support via community dialect (`kysely-neon`).
- **Edge runtime:** supported.
- **Migrations:** no built-in migration system; bring your own (Atlas, `node-pg-migrate`, raw SQL).
- **API style:** SQL-shaped query builder (`db.selectFrom('users').where('id', '=', id).executeTakeFirst()`).
- **Types:** **database-first** — types are declared by hand in a `Database` interface, or generated from an existing DB schema via `kysely-codegen`. There is no schema-as-source-of-truth.
- **Weekly downloads:** ~500K and growing.
- **License:** MIT.

## Comparative Analysis

| Criterion | Drizzle | Prisma | Kysely |
|---|---|---|---|
| **Bundle size (edge function)** | **~7 KB** | ~30 KB (with Driver Adapter) | ~50 KB |
| **Cold-start overhead** | **Negligible** | Low (post Driver Adapters) | Low |
| **Neon serverless driver** | **Native** | Via Driver Adapter (stable) | Community dialect |
| **Edge runtime out of the box** | **Yes** | Yes (with config) | Yes |
| **Schema source of truth** | TS file | `.prisma` DSL | None (hand-written types or codegen) |
| **Migration workflow** | **`drizzle-kit` (generate + apply)** | `prisma migrate` (generate + apply) | External tool required |
| **Codegen step required** | No | **Yes (`prisma generate`)** | Optional (`kysely-codegen`) |
| **SQL transparency** | **High** (queries read like SQL) | Low (hides SQL by default) | **High** |
| **Raw SQL escape hatch** | `sql\`\`` template | `$queryRaw` | `sql\`\`` template |
| **Type inference from schema** | **Automatic** | Generated client | Manual / codegen |
| **Maturity** | Younger (2022) | Most mature (2018) | Younger (2022) |
| **License** | Apache 2.0 | Apache 2.0 | MIT |

## Decision

**Drizzle ORM** is selected as the database client.

Rationale:

1. **The repository layer is already defined as a "SQL resolver."** Drizzle's query API reads like the SQL it produces, which matches the convention in `src/repositories/CLAUDE.md` better than Prisma's active-record style. There is no impedance mismatch between the layer convention and the tool.
2. **Lowest cold-start cost on Neon serverless.** With a ~7 KB footprint and native `@neondatabase/serverless` support, Drizzle is the lightest of the three on the exact runtime profile the project targets (dormant projects, HTTP driver, edge-compatible middleware). On a multi-site portfolio where most projects idle, this advantage compounds.
3. **Schema is the source of truth — no codegen step to keep in sync.** Drizzle's TS schema doubles as types; there is no `prisma generate` to run after every change, and no `Database` interface to hand-maintain like Kysely. This removes a class of "I forgot to regenerate" bugs and integrates cleanly with the existing `sync-validator` agent.
4. **Migration story is built in and versioned in Git.** `drizzle-kit generate` writes timestamped SQL into `scripts/migrations/` — the same directory the existing `/migration` skill already points at. The skill needs minor edits, not a rewrite. Kysely would require choosing and integrating a separate migration tool; Prisma owns the directory and naming, which would conflict with the current skill.
5. **Prisma's Driver Adapter path works, but adds a build step and a heavier client for a marginal gain.** The Prisma 2026 TypeScript-client preview narrows the gap, but it is still a *preview* — committing the template to it is a longevity bet we don't need to take when Drizzle already covers the use case in stable form.

Kysely was the runner-up. It would be the right choice if we wanted to drive an existing database (DB-first) rather than evolve schema from code. Since this is a greenfield template where schema-as-code is preferred, Drizzle wins.

Prisma would be the right choice for a project where the team values pre-built admin tooling (Prisma Studio) and Active-Record ergonomics over SQL transparency. Neither matches the layer conventions of this template.

## Consequences

### Positive

- Repository code reads close to SQL — aligns with the existing `src/repositories/CLAUDE.md` convention with minimal rule changes.
- Schema lives in `src/lib/db/schema.ts` as TypeScript; entities can re-export inferred types instead of being maintained in parallel.
- Migrations are generated automatically from schema diffs and committed to `scripts/migrations/`, keeping the `/migration` skill workflow intact.
- Edge runtime works out of the box for middleware and Better Auth handlers (ADR-004).
- No codegen step in CI; type checking is the only step that needs the schema.

### Negative / Trade-offs

- **Smaller ecosystem than Prisma.** Fewer third-party integrations (admin panels, visualizers). Mitigation: `drizzle-studio` covers the basics; we don't ship admin tooling in this template.
- **Younger project** than Prisma (2022 vs. 2018). Mitigation: backed by a growing user base and the Drizzle team has shipped consistently through 2025–2026. If abandoned, schema is plain TS and migrations are plain SQL — exit cost is low.
- **No automatic seed tooling** equivalent to `prisma db seed`. Mitigation: add a `scripts/seed.ts` script when a project needs it.
- **Repository convention needs a small edit.** The current rule mentions Supabase's `.maybeSingle()`; this becomes `.limit(1)` + `[0] ?? null` in Drizzle. Documented in the Fase 1 migration plan.

### Revisit Triggers

- A project needs Prisma-specific tooling (Prisma Studio for non-technical users, Prisma Accelerate connection pooling at scale) — adopt Prisma for that project only.
- Drizzle's release cadence slows materially or the project shows abandonware signals — re-evaluate against Kysely (lower migration cost than Prisma).
- A team member joins who is significantly more productive in a different ORM — re-evaluate for new projects, not retroactively.
