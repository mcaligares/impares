# ADR-010: Storage Upload Pattern

## Status

Accepted — June 2026

## Context

Per ADR-003, object storage is Cloudflare R2 (S3-compatible API). We need to choose how user-uploaded files reach the bucket. There are three architectural patterns:

1. **Server-side upload through a Next.js Server Action** — the client sends bytes to our backend, our backend streams them to R2.
2. **Client-side direct upload via a pre-signed URL** — the backend issues a short-lived presigned URL; the client `PUT`s the bytes straight to R2.
3. **Hybrid worker** — a Vercel function fronts uploads with custom logic (virus scan, transcoding) before forwarding to R2.

The decision affects Vercel function bandwidth (ADR-001), client UX (progress, chunked uploads), and the security model.

## Decision Drivers

- **Vercel function bandwidth cost** — anything routed through the function counts against the included quota.
- **Vercel function payload limits** — 4.5 MB request body limit on Hobby/Pro for standard routes; Server Actions cap around 1 MB by default.
- **Client UX** — progress, resumability, chunked uploads on large files.
- **Validation** — MIME type and size must be enforced server-side regardless of pattern.
- **Operational complexity** — fewer moving parts is better.

## Options Considered

### Option A — Client-side direct upload via presigned URL

- Client requests a presigned URL from a Server Action (`requestUploadUrl(scope, mime, size)`).
- Server validates MIME and size against `storage.config.ts`, then calls R2's `getSignedUrl` with a short TTL and content-type/length constraints.
- Client `PUT`s the file directly to R2 from the browser.
- Vercel function only sees the metadata request, never the file bytes.

### Option B — Server-side upload through Server Action

- Client posts the file (FormData) to a Server Action.
- Server validates, then streams to R2 with the AWS SDK.
- Server returns the resulting key.

### Option C — Hybrid worker (pre-flight + transform)

- Client uploads to a Vercel function or a Cloudflare Worker.
- Function runs custom logic (virus scan, image resize) and forwards to R2.

## Comparative Analysis

| Criterion | Presigned (client→R2) | Server Action (client→Vercel→R2) | Hybrid worker |
|---|---|---|---|
| **Vercel bandwidth** | **Zero (metadata only)** | Counts against quota | Counts against quota |
| **Max file size** | **Bounded only by R2 (5 GB single PUT)** | ~1 MB (Server Action default) / ~4.5 MB route | Depends on worker |
| **Server-side validation** | MIME/size constraints baked into the presigned URL | Direct in handler | Direct in worker |
| **Progress / chunked uploads** | **Native (browser)** | Requires custom streaming | Requires custom streaming |
| **Implementation cost** | Low | Low | High |
| **Extra services needed** | None | None | Worker runtime |

## Decision

**Client-side direct upload via presigned URL** is selected as the default pattern.

Rationale:

1. **Vercel bandwidth is conserved.** ADR-001 caps included bandwidth at 1 TB/month. Routing user uploads through Vercel burns that quota for no gain. Presigned URLs keep the file traffic entirely between the browser and R2 (which has zero egress per ADR-003 — and ingress is free everywhere).
2. **File size is bounded by R2, not by Vercel.** Server Actions cap at ~1 MB by default; standard routes at 4.5 MB. Presigned PUTs go up to R2's single-request limit (5 GB) with no special handling.
3. **Server-side validation is preserved.** The presigned URL is generated by a Server Action that validates the MIME and size **before** signing; R2 enforces the same content-type and content-length on the actual PUT via the signed constraints. The client cannot bypass either check.
4. **Browser-native progress and resumability** come for free. The client uses standard `fetch` with a streaming body and tracks progress on the request itself.
5. **No additional services.** The hybrid worker pattern only earns its complexity when there's a transformation step (virus scan, transcoding). The template does not need that.

The hybrid pattern remains a project-specific escalation when transformation is required.

## Consequences

### Positive

- Vercel function bandwidth is reserved for application traffic, not user file payloads.
- The action layer is thin: validate, sign, return URL + key. No streaming logic.
- Large files (up to 5 GB) work without custom chunking.
- Browser handles upload progress UI natively.

### Negative / Trade-offs

- **Two round-trips per upload** (request presigned URL → upload). One extra HTTP request per file. Acceptable; total latency is negligible compared to upload time.
- **Client must handle upload errors and retries.** Mitigation: a small client helper in `src/lib/r2/upload-client.ts` wraps the two-step flow with sane defaults.
- **CORS must be configured on the R2 bucket.** Mitigation: a one-time setup documented in `README.md`.
- **Object metadata recorded after upload.** The presigned-URL step issues the key, then the client confirms by calling a follow-up action that writes the `storage` table row. Mitigation: idempotent insert; orphaned uploads cleaned by a periodic cron job (ADR-005).

### Revisit Triggers

- A project needs server-side processing during upload (virus scan, image resize, MIME-deep inspection) → adopt the hybrid worker pattern for that project.
- A project needs uploads larger than 5 GB → switch to R2 multipart uploads (supported by the same presigned-URL flow with `CreateMultipartUpload`).
- CORS configuration becomes a recurring source of bugs → consider proxying via a Cloudflare Worker on the same domain.
